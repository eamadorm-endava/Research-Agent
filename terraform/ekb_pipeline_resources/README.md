# Enterprise Knowledge Base (EKB) Pipeline Resources

### Overview

This directory contains the Terraform configuration for the EKB Pipeline orchestrator.

This module manages API enablement, the service account used by the EKB Pipeline service, Cloud Tasks queues, required IAM permissions across buckets and BQ datasets, and the Cloud Run deployment for the orchestrator.

## Usage: Local Impersonation Guide

To test services locally using the Service Account's identity, developers in the permitted group can use GCP Service Account Impersonation.

### 1. Login with your personal credentials
Ensure you are authenticated with your corporate account:

```bash
gcloud auth login
```
 
### 2. Configure Application Default Credentials (ADC)
Run the following command to allow your local environment to "act as" the Service Account. Replace the email with the one generated by this module:

```bash
gcloud auth application-default login \
    --impersonate-service-account=[SA_NAME]@[PROJECT_ID].iam.gserviceaccount.com
```

## IAM Architecture

The module implements a two-tier IAM structure:

- **Identity Permissions (Inward)**: The `developers_group_email` is granted `roles/iam.serviceAccountTokenCreator` on the Service Account resource. This enables the gcloud impersonation logic.
- **Resource Permissions (Outward)**: The Service Account is granted specific functional roles (e.g., `roles/storage.objectAdmin`, `roles/bigquery.dataEditor`, etc.) across the target projects to perform its automated tasks.

## CI/CD Workflow

This project uses Google Cloud Build for automated deployments:

1. **Feature Branches:** Create a branch for your changes (e.g., `feature/add-sa-roles`).
2. **Pull Request:** Opening a PR to `main` triggers a `terraform plan`.
    - View the results in the GitHub "Checks" tab or GCP Cloud Build history.
    - The PR cannot be merged if the plan fails.
    - The PR requires at least one approver.
3. **Merge to Main:** Merging the PR triggers `terraform apply`.
   - Ensure the plan was reviewed before merging.

## Usage

To enable services, define the `project_services` variable in your `terraform.tfvars` file. To create service accounts, define the service account names and IAM role mappings.

Example:

```hcl
# project_id and main_region must be passed via -var flags in CI/CD

developers_group_email = "your-devs-group@domain.com"

project_services = [
  "run.googleapis.com",
  "dlp.googleapis.com",
  "aiplatform.googleapis.com",
  "bigqueryconnection.googleapis.com",
  "cloudtasks.googleapis.com"
]

ekb_pipeline_service_account_name = "ekb-pipeline-sa"

ekb_pipeline_iam_project_roles = [
  "roles/dlp.admin",
  "roles/aiplatform.user",
  "roles/bigquery.jobUser",
  "roles/logging.logWriter"
]

artifact_registry_name               = "mcp-servers"
ekb_pipeline_cloud_run_name          = "ekb-pipeline"
ekb_pipeline_cloud_run_image_tag     = "latest"
```

## Variables

| Name | Description | Type | Default | Required |
|------|-------------|------|---------|:--------:|
| `project_id` | The ID of the project where resources will be created. (MUST be passed dynamically) | `string` | n/a | yes |
| `main_region` | The main region of the project. (MUST be passed dynamically) | `string` | n/a | yes |
| `developers_group_email` | The email of the Google Group that will be granted the Service Account User role. | `string` | n/a | yes |
| `project_services` | A list of Service APIs to enable. | `list(string)` | `[]` | no |
| `ekb_pipeline_service_account_name` | The name of the service account. | `string` | `ekb-pipeline-sa` | no |
| `ekb_pipeline_iam_project_roles` | A list of roles to be assigned to the service account. | `list(string)` | `[]` | no |
