### Why this refactor is needed
The system previously hardcoded `GoogleAuthConfig` across the `AgentBuilder` and MCP Server Configurations, making it impossible to easily integrate new MCP servers that rely on different OAuth providers (such as the upcoming Microsoft OneDrive integration).

### What has been done
1. **Abstracted OAuth configurations:** Extracted `BaseOAuthConfig`, `GoogleAuthConfig`, and `MicrosoftAuthConfig` into a dedicated `oauth_settings.py` file to clean up the agent settings.
2. **Decoupled MCP settings:** Updated `BaseMCPConfig` to use generic `OAUTH_CONFIG` and `GEMINI_AUTH_ID` attributes instead of hardcoding `GEMINI_GOOGLE_AUTH_ID`.
3. **Updated Factory & Builder:** Removed `auth_config` references from `AgentBuilder` initialization. `MCPToolsetBuilder` now dynamically resolves auth endpoints and scopes directly from individual `mcp_config.OAUTH_CONFIG` schemas.
4. **Updated Auth ID extraction:** `mcp_factory.py` now maps `GEMINI_AUTH_ID` dynamically rather than relying on a Google-specific key.
5. **Fixed all tests:** Validated that both configurations and factories function correctly without `GoogleAuthConfig` dependency injection.

### How to use it now
When adding a new MCP server (e.g., OneDrive):
1. Create its config subclassing `BaseMCPConfig`.
2. Define the exact scope list using an enum (e.g., `OneDriveScopes`).
3. Inject the appropriate provider in `OAUTH_CONFIG` (e.g., `MICROSOFT_AUTH_CONFIG`).
4. Set the `GEMINI_AUTH_ID` through an alias like `ONEDRIVE_AUTH_ID`.
5. The `AgentBuilder` will now seamlessly mount it alongside any Google MCP servers and inject the proper delegated token in Gemini Enterprise production environments.
